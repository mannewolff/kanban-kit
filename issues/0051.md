## Kontext
Die `/roles`-Ansicht (Frontend, #53) soll die Rechte-Matrix als Checkbox-Grid zeigen — und zwar aus **einer Quelle der Wahrheit**, nicht aus einer im Frontend doppelt gepflegten Liste. Auch 2.0 (konfigurierbare Rollen) dockt hier an.

## Aufgabe
- Neuer Read-Endpoint `GET /api/roles/matrix` (Session-Auth), liefert die geseedete Matrix:
  ```json
  {
    "roles": ["VIEWER","MEMBER","ADMIN","OWNER"],
    "permissions": [{ "key": "BOARD_CREATE", "resource": "BOARD", "operation": "CREATE" }, ...],
    "grants": { "VIEWER": ["BOARD_READ", ...], "MEMBER": [...], "ADMIN": [...], "OWNER": [...] }
  }
  ```
- `resource`/`operation` aus dem Permission-Key ableiten (bzw. am Enum hinterlegen); Reihenfolge stabil (Ressourcen-Gruppen, dann C/R/U/D).
- Grants aus `role_permission` (bestehendes `RolePermissionRepository`/Adapter erweitern, falls nötig).

## Akzeptanzkriterium
- `GET /api/roles/matrix` gibt Rollen, alle Permissions (mit resource/operation) und die Grants je Rolle zurück — exakt passend zum `V4`-Seed.
- Stabile, gruppierte Reihenfolge der Permissions.
- IT gegen den Seed; `mvn verify` grün.

## Abhängigkeiten
#48 (finales Permission-Set + Seed).
