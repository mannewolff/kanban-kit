## Kontext
kanban-kit hat heute keine einheitliche Versionszählung: `pom.xml` (`0.1.0-SNAPSHOT`) und `frontend/package.json` (`0.1.0`) laufen unabhängig, es gibt keine `RELEASING.md`, keine Git-Tags. Die Skills `push-main` und `merge-production` sind bereits darauf vorbereitet, eine `RELEASING.md` im Repo-Root zu lesen und deren Release-Schritte auszuführen ("Führe die dort unter dem Push-Trigger beschriebenen Release-Schritte aus"), tun das heute aber mangels Datei nicht — keine Änderung an den Skills selbst nötig. Gewünschtes Schema: X.Y.Z — Z steigt bei jedem `push main`, Y bei jedem `merge production` (Z→0), X nur auf explizite Anordnung "Major-Version erhöhen" im Chat (Y,Z→0).

## Aufgabe
- Neue Datei `VERSION` (Repo-Root, Plaintext) als einzige Quelle der Wahrheit, initial `0.1.0`.
- Neues Skript `scripts/bump-version.mjs` (Node, zero-dep, Aufruf `node scripts/bump-version.mjs patch|minor|major`):
  - liest `VERSION`, berechnet neue Version mit Reset-Semantik (patch: Z+1; minor: Y+1,Z=0; major: X+1,Y=0,Z=0)
  - schreibt `VERSION`
  - ersetzt die Projekt-Version in `pom.xml` per gezielter Regex auf den `<artifactId>manban</artifactId>`-Block (kein `mvn versions:set` — das `versions-maven-plugin` ist lokal nicht gecacht, ein Netzwerk-Download würde den Bump-Schritt fragil machen)
  - ruft `npm --prefix frontend version <neue-version> --no-git-tag-version --allow-same-version` auf (aktualisiert `package.json` + `package-lock.json` konsistent)
  - gibt `Version: X.Y.Z -> X'.Y'.Z'` auf stdout aus
- `pom.xml`: `<version>0.1.0-SNAPSHOT</version>` → `<version>0.1.0</version>` (einmaliger Fix; kanban-kit macht keine klassischen Maven-Releases, der SNAPSHOT-Suffix entfällt dauerhaft).
- Neue Datei `RELEASING.md` (Repo-Root) mit den Abschnitten:
  - `## push main` → `node scripts/bump-version.mjs patch`
  - `## merge production` → `node scripts/bump-version.mjs minor`
  - `## Major-Version erhöhen` → `node scripts/bump-version.mjs major`, mit explizitem Hinweis: dieser Abschnitt wird nur ausgeführt, wenn Manne genau die Phrase "Major-Version erhöhen" im Chat tippt — kein automatischer Trigger, nicht Teil von push/merge.

## Akzeptanzkriterium
- `node scripts/bump-version.mjs patch` von `0.1.0` aus → `VERSION`, `pom.xml`, `frontend/package.json`, `frontend/package-lock.json` zeigen `0.1.1`.
- Anschließend `node scripts/bump-version.mjs minor` → `0.2.0` (Z auf 0 zurückgesetzt).
- Anschließend `node scripts/bump-version.mjs major` → `1.0.0` (Y und Z auf 0 zurückgesetzt).
- `mvn -o -q validate` nach jedem Bump → `pom.xml` bleibt valides XML (kein Strukturbruch durch die Regex).
- `node -e "JSON.parse(require('fs').readFileSync('frontend/package.json'))"` nach jedem Bump → valides JSON.
- Am Ende der Verifikation `VERSION`/`pom.xml`/`package.json`/`package-lock.json` wieder auf `0.1.0` zurücksetzen (die drei Testläufe sind kein echter Release-Sprung).
- **Lauffähig ohne weiteren Repo-Kontext**: `scripts/bump-version.mjs` verwendet ausschließlich Node-Builtins und den `npm`-Unterprozess, keine zusätzlichen npm-Pakete.
