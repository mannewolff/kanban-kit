## Kontext
Kern der eigenen Auth. Ersetzt den Keycloak-Resource-Server.

## Aufgabe
- `POST /api/auth/login` (E-Mail + Passwort), Session als HttpOnly-Secure-SameSite-Cookie (bevorzugt) mit serverseitig prüfbarem Token.
- `POST /api/auth/logout`.
- `GET /api/me` (Claims + Plattform-Rolle + Projekt-Mitgliedschaften).
- `SecurityConfig`: stateless bzgl. Server-Session-Store oder signiertes Cookie, CSRF-Strategie passend zum Cookie-Ansatz, Default-Deny für `/api/**`.
- 2FA-Vorbereitung: Login-Flow so strukturieren, dass ein optionaler zweiter Faktor (TOTP/WebAuthn) später eingehängt werden kann (z. B. Login liefert bei aktivem MFA ein Challenge-Ergebnis statt Session).

## Akzeptanzkriterium
- Login mit korrekten Daten setzt Cookie, `GET /api/me` liefert den User.
- Falsche Daten → 401; unverifizierte User → 403.
- Logout invalidiert.
- Tests inkl. Security-Slice.
- `mvn verify` grün.

### Hinweise
- CSRF-Strategie: Synchronizer-Token bewusst deaktiviert. Begründung: kein Server-Session-Store (STATELESS) und das Auth-Cookie ist HttpOnly + SameSite=Strict → wird nie cross-site gesendet, daher CSRF-resistent. Falls später ein Lax-Cookie o. Ä. gewünscht ist, Double-Submit-Token nachrüsten.
- Logout = Cookie-Löschung (clientseitig). Ein zustandsloses signiertes Token bleibt bis zum Ablauf kryptografisch gültig; echte serverseitige Sofort-Invalidierung (Denylist oder token-version-Spalte) ist eine bewusste spätere Erweiterung.
- session-secret hat einen Dev-Default; in Produktion zwingend MANBAN_SESSION_SECRET setzen (stabil über Neustarts/Instanzen). cookie-secure default true (hinter Caddy-TLS); nur für lokalen Klartext-HTTP-Dev auf false.
- /api/me-Mitgliedschaften sind bis P1/P2 leer (Reader liest project_membership, das noch nicht befüllt wird).
- lokal: Testcontainers braucht DOCKER_HOST (Colima-Socket); api.version-Fix steckt portabel im pom.
