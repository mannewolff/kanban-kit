## Kontext
Die Mail-Architektur ist fertig (3 Ports + JavaMail-Adapter für Verifikation, Passwort-Reset, Einladung; schaltbar über `manban.mail.enabled`, Links über `MANBAN_BASE_URL`) — aber der echte SMTP-Versand funktioniert so nicht: (1) `spring.mail.properties` fehlen komplett (ohne `mail.smtp.auth` + `starttls` scheitert Strato:587), (2) docker-compose reicht `MANBAN_MAIL_ENABLED`/`MANBAN_MAIL_FROM` nicht durch — selbst mit korrekten Credentials bliebe der Versand aus, (3) der Sendepfad (`mailSender.send()`) wurde nie end-to-end getestet. Freigegebener Plan vom 2026-07-11; Entscheidungen: Mailpit-IT ja, Versand bleibt synchron (10s-Timeouts), 587/STARTTLS als Default (465/SSL nur dokumentiert), `SMTP_TO` entfällt (manban adressiert immer den betroffenen Nutzer, kein Sammelpostfach).

Ziel-Zugangsdaten (Strato, `info@mwolff.org`) — echtes Passwort NUR in der gitignorten `.env`:

```
SMTP_HOST=smtp.strato.de   → MANBAN_SMTP_HOST
SMTP_PORT=587              → MANBAN_SMTP_PORT   (587 = STARTTLS, empfohlen; 465 = SSL-Alternative)
SMTP_USER=info@mwolff.org  → MANBAN_SMTP_USER
SMTP_PASS=***              → MANBAN_SMTP_PASSWORD
SMTP_FROM=info@mwolff.org  → MANBAN_MAIL_FROM
(SMTP_TO entfällt — kein fester Empfänger)
```

## Aufgabe
- **TDD zuerst — neue `SmtpMailIT`** (Testcontainers-Muster wie die bestehenden 87 IT): Mailpit-Container als SMTP-Empfänger; `manban.mail.enabled=true` + SMTP-Props via `@DynamicPropertySource` (Auth/STARTTLS aus, Mailpit ist plain). Fälle: (a) Registrierung über die echte API → Mail landet in Mailpit (From/To/Verifikations-Link im Body via Mailpit-REST-API asserten), (b) Passwort-Reset analog, (c) Negativprobe: SMTP-Fehler → Registrierung antwortet mit Fehler, kein User-Zombie (Rollback).
- `src/main/resources/application.yml`: `spring.mail.properties` ergänzen — `mail.smtp.auth: ${MANBAN_SMTP_AUTH:true}`, `mail.smtp.starttls.enable/required: ${MANBAN_SMTP_STARTTLS:true}`, `connectiontimeout/timeout/writetimeout: 10000`. Veralteten „Issue A1"-Kommentar bereinigen.
- `docker-compose.yml` (`manban-api`-Environment): `MANBAN_MAIL_ENABLED`, `MANBAN_MAIL_FROM`, `MANBAN_SMTP_AUTH`, `MANBAN_SMTP_STARTTLS` mit Defaults durchreichen.
- `.env.example`: Mail-Block vervollständigen (Strato-Beispielwerte laut Tabelle oben, `MANBAN_MAIL_ENABLED=true` für Produktion, 465/SSL-Alternative als Kommentar).
- README: kurzer Abschnitt „Mailversand aktivieren" (Env-Block, 3–5 Zeilen).
- Bewusst NICHT: keine neue Abstraktion, kein Async/Outbox, kein HTML-Template (YAGNI; bei Bedarf eigene Issues).

## Akzeptanzkriterium
- `mvn verify` grün inkl. neuer `SmtpMailIT` (Mailpit beweist den kompletten Sendepfad); alle bestehenden Gates unverändert grün.
- Dev-Verhalten unverändert: Default `enabled=false` → Link wird nur geloggt, kein SMTP nötig.
- Manuelle Abnahme durch den Menschen (echter Strato-Kontakt): echte Credentials in gitignorter `.env` auf dem Zielsystem, `docker compose up -d`, Test-Registrierung → Mail kommt an, Verifikations-Link funktioniert. Erst danach Done.

## Abhängigkeiten
Keine. (Der TLS-Handshake mit Strato ist bewusst Teil der manuellen Abnahme, nicht der IT.)
