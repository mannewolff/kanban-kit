## Kontext
`PlatformRole.ADMIN` existiert im Modell (`app_user.platform_role`), wird aber nirgends geprüft — ein Plattform-Admin kann heute nichts, was ein normaler USER nicht auch kann. Entscheidung: Ein Plattform-Admin darf **alles auf allen Projekten** (Super-User); OWNER bleibt auf seine eigenen Projekte beschränkt. Projekt-RBAC (OWNER/ADMIN/MEMBER/VIEWER via `role_permission` + `PermissionChecker`) bleibt unverändert für Nicht-Admins.

## Aufgabe
- `project/application/PermissionChecker`: für einen Aufrufer mit `PlatformRole.ADMIN` **jede** `require(userId, projectId, permission)`- und `requireMembership`-Prüfung passieren lassen (Bypass), unabhängig von `project_membership`. Dazu die Plattform-Rolle des Users laden (`AppUserRepository`/geeigneter Port); Nicht-Admins laufen wie bisher über die Matrix.
- Projekt-Listing: ein Admin sieht **alle** Projekte (`GET /api/projects` liefert für Admins den gesamten Bestand, nicht nur eigene Mitgliedschaften). Für Nicht-Mitglieds-Projekte eine sinnvolle `role`-Kennzeichnung (z. B. Membership-Rolle falls vorhanden, sonst Marker; das Frontend leitet Vollzugriff ohnehin aus `platformRole==='ADMIN'` ab — siehe Frontend-Issue).
- Keine Änderung an den Projekt-Rollen-Rechten für normale User.
- Tests (Testcontainers/MockMvc): Admin kann in einem Projekt agieren (Karte anlegen/verschieben, Board bearbeiten), in dem er **kein** Mitglied ist; Nicht-Admin bleibt bei 403/404; Admin sieht alle Projekte im Listing.

## Akzeptanzkriterium
- `mvn verify` grün inkl. neuer Admin-Bypass-ITs.
- Ein Plattform-Admin kann auf einem fremden Projekt alle Aktionen ausführen; ein normaler User nicht.
- `GET /api/projects` gibt einem Admin alle Projekte zurück.
