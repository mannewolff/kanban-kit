## Kontext
[CLAUDE-java.md](CLAUDE-java.md) §6.3 fordert einen globalen Fehler-Handler via `@RestControllerAdvice` — der referenzierte `GlobalExceptionHandler` stammt aber aus der Toolbox und existiert in manban nicht. Real läuft Fehlerbehandlung über `@ResponseStatus`-Exceptions (verstreut, uneinheitliches Fehlerformat). Stand der Technik ist **RFC 9457 Problem Details** (`application/problem+json`), das Spring nativ unterstützt (`ProblemDetail`, `ResponseEntityExceptionHandler`). Das Frontend (`api/client.ts` mit `ApiError`) profitiert von einem konsistenten Fehler-Body.

## Aufgabe
- TDD: zuerst Tests (Slice-/Unit-Tests für den Handler; je Fehlerklasse Statuscode + Problem-Detail-Felder), dann Implementierung.
- Neuen `common/web/GlobalExceptionHandler` (`@RestControllerAdvice`) einführen, der die bestehenden Domänen-Exceptions auf `ProblemDetail` mappt (Statuscodes wie bisher über die `@ResponseStatus`-Semantik — Verhalten der API-Statuscodes bleibt unverändert, nur der Body wird einheitlich).
- Bean-Validation-Fehler (`MethodArgumentNotValidException`) auf ProblemDetail mit `fieldErrors`-Extension mappen.
- Keine Stacktraces/internen Details nach außen (CLAUDE-security.md Grundprinzip 6).
- Frontend: `ApiError`-Parsing in `frontend/src/api/client.ts` auf das ProblemDetail-Format nachziehen (Feld `detail`/`title`, `fieldErrors`), bestehende UI-Fehlertexte bleiben funktionsfähig; TS-Typen synchron halten.
- Betroffene ITs auf das neue Fehler-Body-Format anpassen (Statuscodes unverändert).
- `CLAUDE-java.md` §6.3: Verweis auf die neue, real existierende Klasse.

## Akzeptanzkriterium
- Alle Fehler-Responses der API liefern `application/problem+json` (RFC 9457); Statuscodes unverändert (bestehende ITs beweisen das).
- Handler zu 100 % unit-/slice-getestet (JaCoCo-Gate ohne neuen Ausschluss); `mvn verify` + `-Ppit` grün; Frontend `npm run build`/`test` grün.
