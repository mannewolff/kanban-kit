## Kontext
Die Admin-Endpunkte (#36) und der Bootstrap (#37) brauchen eine Bedienoberfläche. Zudem muss das Frontend Plattform-Admins Vollzugriff einräumen (alle Projekte, alle Aktionen), passend zum Backend-Super-User (#35).

## Aufgabe
- `frontend/src/api/admin.ts`: `listUsers`, `setPlatformRole(id, role)`, `bootstrap(token)`.
- `frontend/src/pages/AdminPage.tsx` (Route `/admin`): Nutzertabelle (E-Mail, Name, Rolle, verifiziert) mit Umschalter ADMIN↔USER; nur für Plattform-Admins erreichbar. Sidebar-Eintrag "Admin" nur wenn `useAuth().user.platformRole === 'ADMIN'`.
- **Bootstrap-Flow:** eine Möglichkeit, den Env-Token einzugeben und sich zum Admin zu heben (z. B. Route `/admin/bootstrap` mit Token-Feld). Nach Erfolg `AuthContext.refresh()`, sodass die Admin-Navigation erscheint. Fehler (409 inert / 403) verständlich anzeigen.
- **Admin-Vollzugriff im UI:** `lib/roles` (canEditCards/canManageBoards/canManageMembers/canManageProject) geben `true`, wenn der eingeloggte Nutzer Plattform-Admin ist — damit ein Admin auf fremden Projekten alle Aktionen sieht. `platformRole` kommt aus `/api/me` (bereits vorhanden); die Rollen-Helfer bekommen dafür den Admin-Kontext (z. B. Parameter oder ein Hook `useCanEdit`).
- Tests: AdminPage listet Nutzer + Rollenumschalter ruft API; Nav-Eintrag nur für Admins; Bootstrap-Formular ruft `bootstrap` und refresht; roles-Helfer geben für Admin true.

## Akzeptanzkriterium
- `npm test` + `npm run build` grün.
- Als Admin: "Admin"-Eintrag sichtbar, Nutzerliste mit Rollenumschalter; als Nicht-Admin nicht sichtbar/erreichbar. Bootstrap-Token-Eingabe hebt den Nutzer zum Admin (auf frischer Instanz). Ein Admin sieht auf fremden Projekten die Bearbeiten-Aktionen.

## Abhängigkeiten
#36 und #37 (Endpunkte). #35 (Backend-Vollzugriff).
