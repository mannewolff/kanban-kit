## Kontext
Beim ersten Prod-Deployment (kanban.mwolff.org) fiel auf: Der **Verifikations-Link** in der Registrierungs-Mail zeigt auf `…/api/auth/verify?token=` → der Endpunkt liefert `void`/HTTP 200 ohne Body → **leere Seite**. Der **Passwort-Reset-Link** zeigt auf `…/api/auth/reset?token=` → dieser Endpunkt ist POST-only → beim Klick `405 Method Not Allowed`. Beide Flows sind damit für echte Nutzer praktisch kaputt (im Deploy nur über einen manuellen Token-Trick lösbar).

Die passenden **Frontend-Seiten existieren bereits und funktionieren**:
- `/verify` → `VerifyNoticePage` löst den Token per `authApi.verify(token)` ein und zeigt „E-Mail bestätigt ✓".
- `/reset` → `ResetPasswordPage` führt den Reset per POST durch.
`docs/betrieb.md` dokumentiert den Link sogar korrekt als `…/verify?token=…`; nur die Mailer im Backend bauen die falsche (API-)URL.

## Aufgabe
- `RegisterUserService` (Konstruktion `verificationUrl`, ~Zeile 68): `baseUrl + "/api/auth/verify?token=" + token` → `baseUrl + "/verify?token=" + token`.
- `RequestPasswordResetService` (Konstruktion `resetUrl`, ~Zeile 53): `baseUrl + "/api/auth/reset?token=" + token` → `baseUrl + "/reset?token=" + token`.
- Tests, die die erzeugte Mail-URL prüfen, anpassen bzw. ergänzen: URL enthält `/verify?token=` bzw. `/reset?token=` und **nicht** `/api/auth`.
- Keine Frontend-Änderung nötig (Seiten + Routen sind vorhanden).

## Akzeptanzkriterium
- Unit/Slice-Test belegt: Verifikations-Mail-URL = `baseUrl + "/verify?token=…"`, Reset-Mail-URL = `baseUrl + "/reset?token=…"`.
- `mvn verify` grün.
- Nach Redeploy manuell: Klick auf den Verifikations-Link zeigt „E-Mail bestätigt ✓"; Klick auf den Reset-Link zeigt das Passwort-Formular (kein 405, keine leere Seite).

## Abhängigkeiten
Keine. Hinweis: Nach Merge muss die Prod-Instanz neu deployt werden (`git pull` + `up -d --build`), damit echte Nutzer die Flows ohne Token-Trick nutzen können.
