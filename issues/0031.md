## Kontext
Aktuell haben die Board-Spalten eine feste Breite (280px) und wachsen in der Höhe nur mit ihren Karten — rechts und unten bleibt leerer Platz. Vorbild (Toolbox/board-ui-Screenshot): die Spalten nehmen die **gesamte Breite** ein (gleichmäßig aufgeteilt) und reichen über die **gesamte Höhe** bis zum unteren Bildschirmrand, unabhängig von der Kartenzahl. Die Listen-/Item-Ansicht (BoardListPage) füllt die Breite bereits — das Board soll sich genauso verhalten.

## Aufgabe
- **Gilt für alle Listen-Ansichten**, nicht nur das Board: die Inhaltsflächen nehmen die volle Breite ein.
  - `frontend/src/pages/ProjectsPage.tsx`: das Karten-Grid spannt die volle Breite (bereits `auto-fill`, aber ggf. Content-Padding/`maxWidth` prüfen, damit kein toter Raum rechts bleibt).
  - `frontend/src/pages/BoardListPage.tsx`, `ProjectBoardsPage.tsx`, `ProjectMembersPage.tsx`, `EpicsPage.tsx`: Breiten-Constraints (`maxWidth`) entfernen/lockern, sodass die Listen die volle Breite nutzen.
- `frontend/src/components/BoardView.tsx`:
  - Spalten von fester Breite auf **gleichmäßig gefüllte Breite** umstellen: statt `width/minWidth 280` ein `flex: 1 1 0` mit sinnvollem `minWidth` (z. B. 240), sodass die Spalten zusammen die volle X-Achse einnehmen. Der äußere Container spannt die volle Breite (`width: 100%`); horizontales Scrollen nur, wenn zu viele Spalten für die Mindestbreite.
  - Spalten auf **volle Höhe**: die Spalten-Paper bekommen eine `minHeight` bis nahe an den unteren Viewport-Rand (z. B. `calc(100vh - <Offset>)`, Offset = AppBar 64px + Content-Padding + Board-Header), sodass sie nicht mit den Items wachsen, sondern die Y-Achse ausfüllen. Der Karten-Bereich (`Stack`) wächst (`flex: 1`) innerhalb der Spalte.
- `frontend/src/pages/BoardPage.tsx`: sicherstellen, dass die umgebenden Container die Höhe durchreichen (kein `height`-Constraint, das die volle Höhe verhindert).
- Prüfen, dass `AppShell`-Content (`component="main"`) die volle verfügbare Breite/Höhe an das Board weitergibt (ggf. Padding am Board-Bereich reduzieren).
- Kein Umbau der D&D-Logik, keine Änderung an Karteninhalten — reine Layout-Anpassung.

## Akzeptanzkriterium
- `npm test` + `npm run build` grün (bestehende BoardView-Tests bleiben grün; Spalten-Testids unverändert).
- In der laufenden App: die Spalten teilen sich die **volle Bildschirmbreite** (kein toter Raum rechts) und reichen über die **volle Höhe** bis unten, auch bei leeren/wenigen Karten.
- Auch **Projektliste, Listen-Ansicht, Boards-Liste, Mitglieder, Epics** nutzen die volle Breite (kein enger, links-ausgerichteter Block mehr).
- Bei sehr vielen Spalten erscheint horizontales Scrollen erst, wenn die Mindestbreite unterschritten würde.

## Abschlussbericht Issue #31

### Änderungen
- `frontend/src/components/BoardView.tsx` — Spalten von fester Breite (280/280) auf `flex:1 1 0` + `minWidth 240`: volle X-Achse, horizontales Scrollen erst unter Mindestbreite. Volle Höhe: Spalten-Stack `alignItems:stretch` + `minHeight calc(100vh - 210px)`, Spalte als flex-column, Kartenbereich `flex:1`.
- `frontend/src/pages/EpicsPage.tsx` — `maxWidth: 640` entfernt (volle Breite).

### Tests und Checks
- `npm test` → 46 Tests grün (Spalten-/Karten-Testids unverändert)
- `npm run build` → grün

### Hinweise
- Projektliste (auto-fill-Grid), Boards-Liste (List), Mitglieder (MUI-Table hat per Default width:100%) und Listen-Ansicht (flex-Zeilen) waren bereits voll breit — die neue AppShell nutzt keinen Container/maxWidth mehr, daher kein weiterer Eingriff nötig.
- Die volle Höhe basiert auf einem Offset-Calc (≈ AppBar + Board-Header = 210px). Bei stark abweichender Header-Höhe (z. B. Umbruch auf sehr schmal) kann minimal Rand/Scroll entstehen — visuelles Feintuning im Docker möglich.
