## Kontext
C1 (Login nur nach Freigabe) + C2 (Einladung = Freigabe) erzeugen zusammen sonst einen Deadlock: Ein per Token eingeladener Fremd-Nutzer muss sich einloggen, um die Einladung anzunehmen (`MembershipService.accept` braucht eine Session), käme aber ohne Freigabe nie rein. Lösung: Wer sich mit einer E-Mail registriert, für die eine **offene** Einladung existiert, wird automatisch freigegeben. Die Modulgrenze bleibt sauber — das auth-Modul kennt project nicht.

## Aufgabe
- Outbound-Port im auth-Modul, z. B. `RegistrationApprovalPolicy` (`boolean shouldAutoApprove(String normalizedEmail)`), mit Default-Impl „nie" im auth-Modul (falls project-Bean fehlt).
- Impl im project-Modul, die per Invitation-Repository prüft, ob eine **offene** (nicht abgelaufen, nicht akzeptiert) Einladung für die E-Mail existiert.
- `RegisterUserService.register` ([auth/application/RegisterUserService.java:49](src/main/java/org/mwolff/manban/auth/application/RegisterUserService.java)): bei `shouldAutoApprove(email)` → User direkt mit `approvedAt=now` anlegen (kein Admin nötig; keine Admin-Mail aus #0098).
- ArchUnit-Regel: auth hängt nicht von project ab (Port-Inversion).
- TDD zuerst.

## Akzeptanzkriterium
- IT: Owner lädt unbekannte E-Mail ein → diese registriert sich → User ist sofort freigegeben, Login funktioniert, Einladung kann angenommen werden.
- IT: Registrierung ohne offene Einladung → bleibt pending (`approved_at null`).
- ArchUnit grün; `mvn verify` grün.
