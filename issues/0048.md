## Kontext
Die Spezifikation (`rollen_rechte.md`) verlangt eine granulare **CRUD-Rechte-Matrix** pro Ressource als Fundament — Voraussetzung dafür, dass in 2.0 zusätzliche Rollen einfach als weitere Zeile mit eigenen Haken angelegt werden können. Das aktuelle Modell ist grob (`CARD_CREATE/MOVE/DELETE`, `COMMENT_CREATE`, `ATTACHMENT_UPLOAD`, `BOARD_CREATE/DELETE`, `COLUMN_EDIT`, …) und trennt Epics/Tickets nicht und kennt kein Read/Update pro Ressource.

## Aufgabe
- `Permission`-Enum auf die granulare Matrix umstellen:
  - `BOARD_CREATE/READ/UPDATE/DELETE`, `EPIC_CREATE/READ/UPDATE/DELETE`, `TICKET_CREATE/READ/UPDATE/DELETE`,
    `COMMENT_CREATE/READ/UPDATE/DELETE`, `ATTACHMENT_CREATE/READ/DELETE`, `CARD_MOVE`,
    Verwaltung: `MEMBER_INVITE`, `MEMBER_REMOVE`, `PROJECT_EDIT`.
  - `COLUMN_EDIT` geht in `BOARD_UPDATE` auf; `PROJECT_DELETE` entfällt als Projekt-Rolle (Projekt-Löschen wird System-Admin, Issue #49).
- Flyway `V4__granular_permissions.sql`: `permission`-Seed + `role_permission` gemäß Matrix neu aufbauen (`DELETE`+`INSERT`; statische Seed-Tabellen). Rollen-Rechte:
  - **VIEWER:** alle `*_READ`.
  - **MEMBER:** + `TICKET_CREATE/UPDATE/DELETE`, `CARD_MOVE`, `EPIC_CREATE/UPDATE/DELETE`, `COMMENT_CREATE`, `COMMENT_UPDATE`, `ATTACHMENT_CREATE`, `ATTACHMENT_DELETE`.
  - **ADMIN:** + `BOARD_CREATE/UPDATE/DELETE`, `COMMENT_DELETE`, `MEMBER_INVITE`, `MEMBER_REMOVE`.
  - **OWNER:** alle Projekt-Rechte (inkl. `PROJECT_EDIT`).
- Call-Sites auf die neuen Rechte umstellen:
  - `CardService`: nach Typ verzweigen — Karten → `TICKET_*`, Epics → `EPIC_*`; Move → `CARD_MOVE`; Update getrennt von Create.
  - `BoardService`: `BOARD_CREATE/UPDATE/DELETE`; alle Spalten-Operationen → `BOARD_UPDATE`.
  - `AttachmentService`: Upload → `ATTACHMENT_CREATE`, Löschen → `ATTACHMENT_DELETE`.
  - `CommentService.create` → `COMMENT_CREATE` (Update/Delete-Regeln kommen in #50).
- `PermissionChecker` bleibt generisch (nur Enum + Seed ändern sich).

## Akzeptanzkriterium
- `mvn verify` grün. Migration läuft sauber.
- ITs pro Rolle gegen die Matrix: Viewer nur lesen; Member CRUD Ticket/Epic + Move + Kommentar anlegen + Anhang C/D; Admin zusätzlich Board/Spalten/Mitglieder; Owner alles.
- Enum-Werte und geseedete `permission`-Keys sind konsistent (kein verwaister Key).

## Abhängigkeiten
Keine.

## Abschlussbericht Issue #48

### Änderungen
- `Permission.java` — granulare CRUD-Matrix (BOARD/EPIC/TICKET je C/R/U/D, COMMENT C/R/U/D, ATTACHMENT C/R/D, CARD_MOVE, MEMBER_INVITE/REMOVE, PROJECT_EDIT/DELETE); COLUMN_EDIT → BOARD_UPDATE.
- `V4__granular_permissions.sql` — permission + role_permission komplett neu geseedet gemäß Matrix.
- `CardService.java` — verzweigt nach Kartentyp (Ticket/Epic) für Create/Update/Delete (neuer Helfer requireCardOp); Move = CARD_MOVE.
- `BoardService.java` — Rename + alle Spalten-Operationen → BOARD_UPDATE.
- `AttachmentService.java` — Upload → ATTACHMENT_CREATE, Delete → ATTACHMENT_DELETE.
- `AccessTokenService.java` — Token-Binding-Gate CARD_CREATE → TICKET_CREATE.
- `RbacMatrixIT.java` — EXPECTED-Matrix auf die granulare Matrix umgestellt (erschöpfender Test).
- `AppUserRepositoryIT.java` — Zähler: OWNER 24, ADMIN 22, MEMBER 16, VIEWER 5.

### Tests und Checks
- mvn verify (volle Regression) → 12 Unit + 82 IT, 0 Fehler. Verhalten aller Rollen unverändert (nur granular abgebildet).

### Hinweise
- PROJECT_DELETE bleibt vorerst OWNER-Recht (Umstellung auf System-Admin in #49, sonst bräche ProjectService.delete).
- COMMENT_UPDATE/DELETE sind geseedet, die Durchsetzung (Update=Autor, Delete=Admin) folgt in #50 — aktuell gilt noch die bisherige requireModifiable-Logik.
- READ-Rechte sind in der Matrix, werden aber weiter über requireMembership durchgesetzt (alle Rollen sind Mitglieder).
