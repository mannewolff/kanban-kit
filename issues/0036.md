## Kontext
Ein Plattform-Admin (siehe #35) muss andere Nutzer verwalten können: alle Nutzer sehen und andere zum Admin ernennen bzw. degradieren. Bisher gibt es dafür keine Endpunkte.

## Aufgabe
- Neues Web-Modul `admin` (oder unter `auth/web`): 
  - `GET /api/admin/users` — Liste aller Nutzer (id, email, displayName, platformRole, emailVerified).
  - `PATCH /api/admin/users/{id}` — setzt `platformRole` (ADMIN|USER). **Last-Admin-Guard:** der letzte verbleibende Admin darf sich/den letzten Admin nicht degradieren (409, analog Last-Owner).
- Admin-Gate: die Admin-Endpunkte nur für `PlatformRole.ADMIN` (Service-Check `requirePlatformAdmin(userId)` → 403 sonst) und nur mit Session-Auth (nicht via PAT). **Wichtig:** als **Service-Check pro Endpunkt**, nicht als pauschale SecurityConfig-Regel auf `/api/admin/**`, damit der Bootstrap-Endpunkt (#37) ausgenommen bleibt.
- Neue Use-Cases im auth-/admin-Application-Layer (listUsers, changePlatformRole) + Ports; Wiederverwendung `AppUserRepository`.
- Tests: Admin listet Nutzer; Admin ernennt/degradiert; Nicht-Admin bekommt 403; letzter Admin nicht degradierbar (409).

## Akzeptanzkriterium
- `mvn verify` grün inkl. neuer ITs.
- Admin kann Nutzerliste abrufen und Plattform-Rollen ändern; Nicht-Admin 403; letzter Admin bleibt Admin (409-Guard).

## Abhängigkeiten
