## Kontext
[CLAUDE-react.md](CLAUDE-react.md) deklariert ESLint als verbindliches Gate vor jedem Push (typescript-eslint, react, react-hooks, jsx-a11y, flat config) — **nichts davon existiert im Projekt**: kein ESLint-Paket, keine `eslint.config.js`, kein `lint`-Script. Konsequenz zuletzt in #0068: Die `exhaustive-deps`-Bereinigung musste ohne Lint-Netz manuell geprüft werden. Auch die CI hat keinen Lint-Step und nutzt `npm install` statt `npm ci`.

## Aufgabe
- ESLint (aktuelle Version, flat config `eslint.config.js`) in `frontend/` einführen mit:
  - `typescript-eslint` (recommended, type-aware wo verhältnismäßig),
  - `eslint-plugin-react` (recommended + jsx-runtime),
  - `eslint-plugin-react-hooks` (recommended, insbes. `exhaustive-deps`),
  - `eslint-plugin-jsx-a11y` (recommended).
- `package.json`: Script `"lint": "eslint src"`.
- Alle auftretenden Findings **beheben** (nicht pauschal deaktivieren). Erwartete Treffer: die zwei begründeten `exhaustive-deps`-Disables aus #0068 (Begründungskommentare existieren bereits — prüfen, dass sie stehen bleiben dürfen). Wo eine Regel nachweislich projektfremd ist: einzeln + begründet in der Config deaktivieren (dokumentiert, analog zum Checkstyle/PMD-Vorgehen im Backend #0075).
- CI ([.github/workflows/ci.yml](.github/workflows/ci.yml)): im Frontend-Job `npm install` → `npm ci` und Step `npm run lint` ergänzen.
- `CLAUDE-react.md`-ESLint-Abschnitt an die echte Config anpassen (koordiniert mit dem Doku-Issue).

## Akzeptanzkriterium
- `cd frontend && npm run lint` läuft grün über `src/`.
- Negativprobe: ein bewusst eingebauter Verstoß (z. B. fehlende Dependency im Dep-Array, `<img>` ohne alt) macht lint rot (lokal, danach zurücknehmen).
- `npm run build` + `npm test` weiter grün; CI-Workflow enthält lint-Step + `npm ci`.

## Abhängigkeiten
Keine. Das Doku-Issue übernimmt danach die reale Config in den ESLint-Abschnitt.
