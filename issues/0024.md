## Kontext
manban hat bisher keine Epics. Die Toolbox löst Epics in derselben Item-Tabelle: `type (ITEM|EPIC)` + selbstreferenzierendes `parent_id` (`ON DELETE SET NULL` → Kinder bleiben beim Epic-Löschen erhalten) + optionales `shortcode`; Epics halten keine Board-Position und liefern Fortschritt (done/total der nicht-archivierten Kinder). Vorlage: `../java/src/main/resources/db/migration/V22__kanban_epics.sql`, `V23__kanban_epic_shortcode.sql`, `../java/src/main/java/org/mwolff/api/kanban/*Epic*`. Das mappt 1:1 auf manbans `card`-Tabelle (board-scoped `number`, `active_position` Generated-Column, `card_dependency`).

## Aufgabe
- Neu `src/main/resources/db/migration/V2__epics.sql` (Postgres): `card` um `type varchar(20) not null default 'CARD'`, `parent_id bigint null` (self-FK `REFERENCES card(id) ON DELETE SET NULL`, Index), `shortcode varchar(16) null` erweitern. Generated-Column `active_position` neu definieren (drop + add): `CASE WHEN archived OR type='EPIC' THEN NULL ELSE position_in_column END` STORED; Unique-Constraint auf `(column_id, active_position)` neu setzen. V1-Baseline NICHT ändern (additive Migration).
- `card`-Domain-Record + JPA-Entity um `type`, `parentId`, `shortcode` erweitern; `CardType`-Enum (CARD|EPIC).
- Neuer `EpicService` (application) bzw. Erweiterung `CardService`: Epic anlegen (type=EPIC, optional shortcode), Karte einem Epic zuordnen/lösen (setzt/löscht `parent_id`, nur type=CARD zuweisbar), Epic-Liste je Board mit Fortschritt (`done`/`total` nicht-archivierter Kinder; `done` = Kind in Done-Spalte), Epic löschen (Kinder bleiben via ON DELETE SET NULL). Move/Reindex in `CardRepositoryAdapter`/`CardService` zählt nur `type='CARD'` (Epics nie in Positions-Namespace).
- `EpicController` (web): `GET /api/boards/{id}/epics` (Liste mit Fortschritt), `POST /api/boards/{id}/cards` um Typ/Kürzel erweitern oder eigener Epic-Anlage-Pfad, `PATCH /api/cards/{id}` um `parentId` (Zuordnung) erweitern, `DELETE` für Epic. RBAC über bestehende Card-Permissions (CARD_CREATE etc.), board-scoped.
- Tests (Testcontainers/MockMvc): Epic-CRUD, Kind-Zuordnung/-Lösung, Fortschritt korrekt, `active_position` bleibt für Epics NULL (keine Kollision), Move ignoriert Epics, Epic-Löschen löst Kinder (bleiben erhalten).

## Akzeptanzkriterium
- `mvn verify` grün inkl. neuer Epic-Integrationstests.
- Migration V2 läuft additiv auf der V1-Baseline (Flyway) ohne Fehler.
- Epic anlegen, Karte zuordnen, Epic-Liste liefert korrekten Fortschritt; Reindex/Move einer Spalte mit Epics kollidiert nicht mit dem Unique-Constraint.
