## Kontext
Alle Registrierten sind Plattform-USER. Es braucht einen Weg, den ersten Admin zu erzeugen, ohne DB-Handgriffe. Entscheidung: ein Einmal-Bootstrap über einen **Env-Token**, der **nur greift, solange es null Plattform-Admins gibt** — sobald ein Admin existiert, ist der Pfad inert (selbstheilend, kein Aussperren).

## Aufgabe
- Konfiguration: `MANBAN_BOOTSTRAP_ADMIN_TOKEN` (application.yml → `manban.bootstrap.admin-token`, eigenes `@ConfigurationProperties`-Record). Leer/nicht gesetzt = Bootstrap deaktiviert.
- Endpunkt `POST /api/admin/bootstrap` (Body `{ token }`): erlaubt für einen **eingeloggten** Nutzer (Session-Auth), NICHT admin-gegated. Logik:
  - wenn bereits ein Plattform-Admin existiert → 409 (inert),
  - wenn kein Token konfiguriert → 403 (deaktiviert),
  - wenn Token nicht übereinstimmt (konstanter Zeitvergleich) → 403,
  - sonst: den aktuell eingeloggten Nutzer auf `PlatformRole.ADMIN` heben und zurückgeben.
- SecurityConfig: `/api/admin/bootstrap` erfordert Authentifizierung (Session), aber NICHT die Admin-Autorität (Chicken-Egg).
- Kein Token im Code/DB; nur Env.
- Tests: erster eingeloggter Nutzer wird mit korrektem Token Admin; falscher Token → 403; kein Token konfiguriert → 403; sobald ein Admin existiert → 409 (auch mit korrektem Token).

## Akzeptanzkriterium
- `mvn verify` grün inkl. Bootstrap-ITs.
- Auf frischer Instanz (kein Admin) macht der korrekte Token den eingeloggten Nutzer zum Admin; danach ist der Endpunkt inert (409).

## Abschlussbericht Issue #37

### Änderungen
- `auth/application/BootstrapService.java` — bootstrap(userId, token): 409 wenn Admin existiert; 403 wenn Token fehlt/falsch (MessageDigest.isEqual, konstante Zeit); sonst eingeloggten Nutzer auf ADMIN heben
- `auth/application/BootstrapProperties.java` — manban.bootstrap.admin-token; `BootstrapUnavailableException` (409), `InvalidBootstrapTokenException` (403)
- `auth/web/BootstrapController.java` — POST /api/admin/bootstrap {token}; unter /api/admin/** (Session-Auth), aber NICHT admin-gegated
- `application.yml` — manban.bootstrap.admin-token: ${MANBAN_BOOTSTRAP_ADMIN_TOKEN:} ; `docker-compose.yml` reicht die Env-Var durch
- Tests: `BootstrapServiceTest` (deaktiviert/falsch/inert/elevate) + `BootstrapIT` (unauth 401, korrekter Token → ADMIN)

### Tests und Checks
- `mvn verify -Dtest=BootstrapServiceTest -Dit.test=BootstrapIT` → BUILD SUCCESS (4 + 2 grün)

### Hinweise
- Deployment: `MANBAN_BOOTSTRAP_ADMIN_TOKEN` setzen, sich einloggen, POST /api/admin/bootstrap mit dem Token → erster Admin. Danach ist der Pfad inert (409). Token danach entfernen empfohlen.
- Der Endpunkt ist bewusst NICHT admin-gegated (sonst könnte niemand der erste Admin werden); Schutz allein über Token + Admin-Existenz.
