## Kontext
Das Backend erzwingt 100 % Line/Branch (JaCoCo, #0072) und 100 % Mutation (PIT, #0073/#0077). Das Frontend hat **kein Coverage-Gate**: `vitest run` läuft ohne Coverage-Messung, Schwellwerte existieren nicht. 86 Tests in 23 Dateien sind vorhanden, aber niemand misst, was sie abdecken. Stand der Technik: Vitest-Coverage (v8-Provider) mit `thresholds` in der Vite-Config — Build bricht bei Unterschreitung.

## Aufgabe
- `@vitest/coverage-v8` als devDependency; Coverage-Konfiguration in `vite.config.ts` (bzw. `vitest.config.ts`): `provider: 'v8'`, `reporter: ['text', 'html']`, `include: ['src/**']`, sinnvolle `exclude` (z. B. `main.tsx`, `vite-env.d.ts`, Test-Setup — einzeln begründet, analog §5.2-Prinzip des Backends).
- Ist-Coverage messen (`npm test -- --coverage` bzw. Script `"test:coverage"`).
- Offensichtliche Lücken mit Tests schließen, wo verhältnismäßig (Fokus: `lib/`-Logikmodule und `api/`-Mapper zuerst — reine Logik, billig testbar).
- `thresholds` (lines/branches/functions/statements) auf den dann erreichten Stand als **ehrliches, grünes Gate** setzen, mit Kommentar „Zielpfad: schrittweise anheben" (analog zum PIT-Vorgehen in #0073). Keine künstlich niedrige Schwelle ohne Zahlbezug.
- CI: Coverage-Lauf in den Frontend-Job aufnehmen (ersetzt oder ergänzt `npm test`).
- `CLAUDE-react.md` §Tests um das Coverage-Gate ergänzen (koordiniert mit dem Doku-Issue).

## Akzeptanzkriterium
- `npm run test:coverage` erzeugt Report und bricht bei Unterschreitung der Schwellen (Negativprobe: Schwelle temporär über Ist setzen → rot; zurücknehmen).
- Gate läuft in CI; `npm run build` + `npm test` weiter grün.
- Schwellwerte + Ausschlüsse stehen begründet in der Config; Ist-Coverage-Zahl im Abschlussbericht.

## Abhängigkeiten
Keine. Sinnvoll nach #0085 (ESLint), damit die CI-Frontend-Pipeline in einem Zug komplett wird.
